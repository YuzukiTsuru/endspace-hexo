## 为什么还在用静态站点

在动态 CMS 和全栈框架盛行的今天，静态站点生成器（SSG）依然有其独特的优势：

- **性能**：纯 HTML，零运行时开销
- **安全**：没有数据库，没有服务端攻击面
- **成本**：任意静态托管即可，几乎免费
- **版本控制**：内容即代码，Git 管理

## 三大选手

### Hexo

- **语言**：Node.js
- **模板**：EJS / Pug / Nunjucks
- **亮点**：中文生态成熟，插件丰富，上手简单
- **痛点**：构建速度慢，大站点生成耗时长

### Hugo

- **语言**：Go
- **模板**：Go Template
- **亮点**：构建速度极快（秒级），单二进制部署
- **痛点**：Go Template 语法不直观，主题生态不如 Hexo 丰富

### Astro

- **语言**：Node.js
- **模板**：任意框架组件（React / Vue / Svelte）
- **亮点**：零 JS 默认输出，岛屿架构，现代化开发体验
- **痛点**：较新，生态还在成长；博客场景略显"杀鸡用牛刀"

## 适合场景

| 场景 | 推荐 | 原因 |
|---|---|---|
| 个人博客 | Hexo | 主题多，插件全，中文文档友好 |
| 大型内容站 | Hugo | 构建速度快，适合数千页面 |
| 技术文档站 | Astro | 组件化开发，交互丰富 |
| 快速原型 | Hugo | 单文件部署，零依赖 |
| 前端开发者 | Astro | 可以用熟悉的框架写组件 |

## 我的选择

我选择 Hexo，原因很简单：

1. Node.js 生态与我日常开发工具链一致
2. 丰富的中文主题和插件资源
3. EJS 模板足够灵活，可以自由定制

构建速度确实是个问题，但对于个人博客的体量来说完全可以接受。如果未来内容量增长到数千篇，可能会考虑迁移到 Hugo。

## 迁移成本

如果你在考虑迁移，需要注意：

- **Hexo → Hugo**：Markdown 基本通用，但 Front Matter 格式和模板需要完全重写
- **Hexo → Astro**：内容可复用，但主题和组件需要用框架重写
- **任何迁移**：自定义插件和脚本大概率需要重写

迁移前建议先在小规模测试站点上试水，不要一上来就动生产环境。
